02 · Objetivos del portal

Para qué existe esta plataforma.

Cuatro propósitos guían cada módulo del portal.

1

Centralizar instrumentos

Concentrar en un solo lugar los instrumentos institucionales del SLEP y todos sus indicadores.

Permitir que la dirección revise y devuelva con observaciones, dejando la traza dentro del mismo sistema.

04 · Decisiones
04 · La decisión clave

Por qué Google Apps Script
y no una base de datos tradicional.

La pregunta
¿Construimos un backend propio con una base de datos y servidor pagado, o aprovechamos el ecosistema Google que ya usa el SLEP a diario?
Costo cero en backend y base de datos
Vive dentro del Workspace institucional
Autenticación con la cuenta del servicio
Datos auditables y exportables
La respuesta
Quedarnos dentro del ecosistema Google del SLEP, sin sumar costos ni dependencias externas.
Apps Script publica una Web App que actúa como API. Google Sheets cumple el rol de base de datos: cada hoja es una tabla con su propio esquema.

04 · Calce institucional
04 · Por qué esta solución sí calza con el SLEP

No es solo una herramienta técnica. Es una decisión institucional.

La plataforma se diseñó para convivir con la operación real del servicio, no para exigir un cambio cultural imposible.

01 · Ecosistema conocido

Vive dentro del Workspace que el servicio ya usa.

No obliga a introducir un proveedor nuevo, ni una operación paralela de usuarios, respaldos o permisos fuera del entorno institucional.

Google Workspace · Continuidad operativa
02 · Costo responsable

Resuelve una necesidad pública sin sumar infraestructura pagada.

La decisión técnica cuida recursos, evita dependencia externa y mantiene el foco en gestión, seguimiento y mejora del servicio.

Costo marginal cero · Sostenibilidad
03 · Trazabilidad útil

El dato sigue siendo visible, exportable y auditable.

La información no se encierra en una caja negra: queda disponible para control interno, revisión directiva y trabajo cotidiano de las áreas.

Gobernanza del dato · Transparencia
La solución calza con el SLEP porque respeta su operación, su entorno tecnológico y la forma en que necesita rendir cuentas.

06 · Operación
06 · Del editor al usuario

Cómo viaja un cambio
desde el equipo hasta producción.

Paso 1
Escribir el cambio
El equipo trabaja en VS Code con apoyo de Copilot y Claude Code.
Paso 2
Subir a GitHub
Cada cambio relevante queda registrado en el repositorio con su historial.
Paso 3
Vercel publica
Vercel toma el código de GitHub y publica automáticamente la nueva versión del frontend.
Paso 4
Backend en Apps Script
Los cambios del backend se copian al proyecto Apps Script y se vuelve a publicar la Web App.
Paso 5
Validar en ambiente real
Se revisa con usuarios clave que el cambio funciona como se esperaba.
Paso 6
Comunicar
Se informa al equipo qué cambió y cómo afecta su operación diaria.

06 · Vercel
06 · Por qué Vercel

El portal se publica
en Vercel, conectado a GitHub.

Cómo funciona
  • Vercel está conectado directamente al repositorio del portal en GitHub.
  • Cada vez que se sube un cambio, Vercel compila el sitio y lo deja disponible para el usuario final.
  • Si una versión presenta problemas, se puede volver a una anterior con un clic.
  • Permite previsualizar cambios antes de publicarlos al equipo completo.
Qué ganamos
  • Despliegue automático, sin pasos manuales propensos a error.
  • HTTPS y rendimiento global incluidos por defecto.
  • Plan gratuito alcanza para el uso institucional actual.
  • Conexión directa con el flujo de GitHub que ya usamos.
Cero costo Cero servidores Rollback inmediato
